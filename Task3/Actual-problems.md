# Актуальные проблемы
- Время ожидания ответа по всем компаниям, как минимум (это если опрос по данным реализован параллельно), равно времени ожидания самого долго партнера. 
- Ошибка выдачи информации на стороне партнера может вызывать проблемы на стороне core-app или ins-comp-settlement
- core-app опрашивет ins-product-aggregator каждые 15 минут, что создает дополнительную нагрузку
- ins-comp-settlement выполняет запрос раз в сутки, что ведет к несогласованности данных с данными core-app
- синхронный запрос core-app и ins-comp-settlement может привести к таймауту, отчего оба процесса будут отклонены
- При увеличение количества страховых компаний партнеров, вероятность того, что хотя бы одна компания будет недоступна во время ночного обновления увеличится
    - В случае сбоя ins-comp-settlement будет работать с устаревшими данными или не будет выполнять взаиморасчет из-за ошибки


# Рекомендации
- перевести текущую архитектуру на event-driven архитектуру
    - ins-product-aggregator - будет выполнять обновление данных асинхронно по тригеру или расписанию
    - core-app подпишется на событие обновления ставок партнеров и будет иметь всегда актуальные данные
    - ins-comp-settlement - подпишется на событие core-app, которое будет сообщать о новом соглашение, и накапливать данные с актуальными на тот момент данными.

    - Данное решение позволит избежать несогласованности между сервисами
    - Ошибками в системе связанными с постоянной подгрузкой данных со стороны партнеров
    - Время ожидания подгрузки актуальных данных сократиться

- Так же стоит использовать Transaction outbox подход для гарантированной доставки сообщений из core-app в ins-comp-settlement
    - Это позволит избежать некорректного реестра в случае потери события
    - Оформление страховки - финансовая операция - данные должны быть гарантированно доставлены
